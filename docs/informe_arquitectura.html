1 Descripción del MVP implementado

StockAI SmartFlow es un sistema de gestión de bodega (WMS, por sus siglas en inglés) orientado a empresas que trabajan con productos perecederos. El núcleo del negocio se sustenta en el algoritmo FEFO (First Expired, First Out): priorizar el despacho de los lotes más próximos a vencer antes que los más recientes.

El MVP implementado cubre el ciclo completo de operación: ingreso de mercadería con identificación automática por visión artificial, control de stock en tiempo real, sistema de alertas automáticas con notificación por correo electrónico, y un panel de monitoreo para los encargados de bodega.

2 Implementación de endpoints REST

La API fue diseñada siguiendo los principios de Roy Fielding para interfaces REST: orientada a recursos, sin estado, con verbos HTTP semánticamente correctos y códigos de respuesta estándar. Todos los endpoints están protegidos con autenticación mediante JWT (JSON Web Tokens) y control de roles.

3 Persistencia de datos

La persistencia se implementó con PostgreSQL 15 como motor relacional y Sequelize 6 como ORM, siguiendo el patrón de repositorio implícito en la capa de servicios. El esquema de la base de datos está completamente versionado mediante migraciones de Sequelize CLI, ejecutadas automáticamente al arrancar el servidor.

3.1 Decisión de diseño: eliminación lógica global

Todos los modelos principales utilizan eliminación lógica (paranoid: true en Sequelize): los registros nunca se borran físicamente. En cambio, se asigna un valor a la columna deleted_at con la fecha de eliminación, y Sequelize filtra automáticamente esos registros en todas las consultas. Esto garantiza trazabilidad completa del historial de inventario con fines regulatorios y de auditoría.

3.2 Seguridad de credenciales

Todas las credenciales de acceso a la base de datos, las claves JWT y la configuración SMTP se almacenan en el archivo backend/.env, excluido del repositorio mediante .gitignore. El repositorio incluye un archivo .env.example con la estructura de variables sin valores reales. Docker Compose inyecta estas variables como variables de entorno del contenedor en tiempo de ejecución.

4 Debate arquitectónico: Monolito Modular vs. Microservicios

"La arquitectura correcta no es la más sofisticada; es la que resuelve el problema actual con el menor costo de mantenimiento futuro. Un MVP que no llega a producción por exceso de complejidad arquitectónica es, técnicamente, un fracaso."

4.1 La decisión adoptada: Monolito Modular con Microservicio de IA

StockAI SmartFlow implementa un Monolito Modular para el núcleo del negocio (Node.js + Express), con un único Microservicio independiente para el componente de visión artificial (Python + FastAPI). Esta arquitectura híbrida no es una concesión, sino una decisión deliberada basada en tres factores concretos.

Factor 1 — Tecnología incompatible: el motor de visión artificial requiere OpenCV, NumPy y bibliotecas de procesamiento de imagen que son nativas del ecosistema Python y no tienen un equivalente de calidad en Node.js. Forzar su integración en el monolito habría implicado el uso de bindings de C++ con rendimiento inestable. La separación en un microservicio permite que cada componente emplee el lenguaje óptimo para su función.

Factor 2 — Escalado independiente: el procesamiento de imágenes es computacionalmente costoso (análisis ORB de 500 puntos clave por imagen) y esporádico (solo durante el ingreso de mercadería). Un microservicio permite escalar exclusivamente ese componente cuando se requiere, sin replicar todo el backend.

Factor 3 — Simplicidad del núcleo: el dominio de negocio central —lotes, FEFO, alertas, usuarios— presenta alta cohesión interna y bajo acoplamiento entre módulos. Fragmentarlo en microservicios a nivel de MVP habría introducido latencia de red, complejidad en transacciones distribuidas y costo operacional sin beneficio real a esta escala.

4.3 Análisis de trade-offs

Ventajas del Monolito Modular (núcleo)

  • Una sola base de código para desplegar y monitorear
  • Transacciones ACID nativas sin coordinación distribuida
  • Latencia cero entre módulos internos (llamadas en memoria)
  • Menor costo operacional en la etapa de MVP
  • Depuración directa con un solo proceso
  • Migraciones de esquema atómicas y versionadas

Limitaciones a mayor escala

  • Escalar requiere replicar todo el servicio, no solo el módulo congestionado
  • Un fallo catastrófico afecta todos los módulos simultáneamente
  • Equipos grandes deben coordinarse en el mismo repositorio
  • Los despliegues afectan toda la aplicación de forma simultánea

Ventajas del Microservicio de IA

  • Lenguaje óptimo para visión artificial (Python / OpenCV)
  • Escalado independiente según la demanda de procesamiento
  • Fallo aislado: si el servicio cae, el inventario sigue operando
  • Despliegue y actualización del modelo sin modificar el backend

Costo del Microservicio

  • Latencia de red en cada identificación por imagen
  • Complejidad adicional de orquestación en Docker
  • Requiere gestión de verificación de salud (health check) independiente
  • Manejo explícito de errores de red y tiempos de espera

4.4 Por qué no se optó por microservicios completos en esta etapa

Una arquitectura de microservicios pura habría implicado separar el módulo de usuarios, el de alertas, el de lotes y el de productos en servicios independientes. A nivel de MVP, esto habría introducido los siguientes costos sin una contrapartida real.

Transacciones distribuidas: la operación de registrar un lote involucra actualizar el stock, verificar el umbral, crear una alerta y registrar la acción en auditoría. En un monolito, todo ocurre en una única transacción ACID. En microservicios, requeriría un patrón Saga o confirmación en dos fases, multiplicando la complejidad sin beneficio para un equipo pequeño.

Latencia interna: las consultas del panel combinan datos de productos, lotes y alertas en tiempo real. En un monolito son consultas JOIN que se resuelven en milisegundos. En microservicios serían múltiples llamadas HTTP sincrónicas o un API Gateway con agregación, añadiendo entre 50 y 200 ms de latencia innecesaria.

Costo operacional: cinco servicios Docker ya representan una complejidad de orquestación manejable. Diez o quince servicios a este nivel de madurez del producto implicarían un costo de operación que superaría al costo de desarrollo del propio sistema.

Veredicto arquitectónico

La arquitectura elegida —Monolito Modular con Microservicio de IA— es la respuesta de ingeniería pragmática correcta para este MVP. El monolito garantiza coherencia transaccional, velocidad de desarrollo y simplicidad operacional donde el dominio es homogéneo. El microservicio de IA es la única excepción justificada por una incompatibilidad tecnológica real, no por aspiraciones de escalabilidad prematura. Esta decisión puede revertirse cuando el sistema alcance una escala que lo justifique: los módulos internos del monolito están suficientemente desacoplados para ser extraídos como servicios independientes en una etapa futura, sin necesidad de reescribir la lógica de negocio.

5 Gestión de datos y trazabilidad

El sistema implementa tres mecanismos de persistencia complementarios que garantizan la integridad y el historial completo de las operaciones.

PostgreSQL (datos del negocio): almacena el catálogo de productos, los lotes con su historial de movimientos, las alertas generadas, los usuarios y todos los registros de auditoría. El esquema está definido mediante migraciones versionadas que se ejecutan automáticamente en cada arranque, garantizando que el estado de la base de datos siempre coincida con la versión del código.

Redis (estado efímero): gestiona el control de tasa de peticiones por IP y por usuario (máximo 100 peticiones por minuto en general, 10 en rutas de autenticación) y la invalidación de tokens de sesión. Si Redis se reinicia, se regenera automáticamente sin pérdida de datos del negocio.

Registro de auditoría (AuditLog): cada operación crítica —creación de lote, consumo, merma, actualización de umbrales— genera un registro inmutable en la tabla audit_logs con el identificador del usuario, la acción ejecutada, el estado anterior y el nuevo estado. Esta tabla no utiliza eliminación lógica, para garantizar que ningún registro pueda ser borrado, ni siquiera de forma accidental.
