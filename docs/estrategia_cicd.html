1 Definición del problema

Sin automatización, cada actualización del sistema StockAI SmartFlow requiere que un desarrollador ejecute manualmente entre ocho y doce comandos en secuencia: construir las tres imágenes Docker, subirlas a un registro de contenedores, conectarse al servidor de producción, aplicar los manifiestos de Kubernetes y verificar que los pods arrancaron correctamente. Una sola equivocación en este proceso puede dejar el sistema de bodega fuera de servicio durante minutos o permitir que una versión defectuosa llegue a producción sin ser detectada.

La estrategia CI/CD propuesta elimina por completo esta intervención manual. Desde el momento en que un desarrollador fusiona código a la rama principal, la cadena de automatización toma el control y solo libera a producción lo que ha sido verificado de forma exitosa.

"El objetivo no es desplegar rápido. Es desplegar con confianza. CI/CD no reemplaza la revisión humana del código; reemplaza la ejecución manual de pasos repetibles que un humano puede omitir o ejecutar en orden incorrecto."

2 Conceptos fundamentales: CI y CD

2.1 Integración Continua (CI)

Es la práctica de integrar cambios de código con frecuencia —idealmente varias veces al día— verificando de manera automática que cada integración no rompe el sistema. En StockAI SmartFlow, CI implica que cada vez que un desarrollador sube código a GitHub, el sistema valida de forma automática que el código compila correctamente y que las imágenes Docker se construyen sin errores.

2.2 Entrega Continua (CD)

Extiende la integración continua llevando automáticamente el código verificado hasta el entorno de producción. En StockAI SmartFlow, CD significa que, una vez que el código supera las validaciones, las imágenes se publican en el registro de contenedores y se despliegan en el clúster de Kubernetes sin intervención humana, salvo por una aprobación explícita antes del paso final.

3 Estrategia de ramas (GitFlow simplificado)

El flujo de trabajo se organiza en torno a dos tipos de ramas con comportamiento diferenciado en el pipeline de automatización.

── Desarrollo de una nueva funcionalidad ────────────────────────── feature/ingreso-manual ──▶ CI ejecuta: lint + construcción de imágenes (sin publicar) │ ▼ Pull Request main ──▶ CI + CD: construcción → publicación → despliegue en Kubernetes ── Cada commit en main representa una versión desplegable ───────── main commit abc1234 ──▶ imagen: stockai-backend:abc1234 main commit def5678 ──▶ imagen: stockai-backend:def5678 Beneficio: rollback inmediato utilizando el tag del commit anterior.

Lint CI Todo push o PR Verifica la sintaxis del código Node.js. Detecta errores básicos antes de construir imágenes. Bloquea el pipeline. No se construye ninguna imagen.
Construcción de imágenes CI Todo push o PR Construye las 3 imágenes Docker (backend, frontend, servicio de IA) usando el stage production del Dockerfile multi-stage. No las publica en el registro. Bloquea el pipeline. Indica un error en el Dockerfile o en las dependencias.
Publicación de imágenes CD Solo en push a main Sube las 3 imágenes al GitHub Container Registry (ghcr.io) con dos etiquetas: el SHA del commit (p. ej., abc1234) y latest. La etiqueta por SHA garantiza trazabilidad exacta. Bloquea el despliegue. Las imágenes previas permanecen en producción.
Aprobación manual Gate Antes del despliegue a producción GitHub Environments pausa el pipeline y notifica al equipo responsable. Un encargado revisa los cambios y aprueba o rechaza el despliegue en la interfaz de GitHub. El pipeline permanece en espera hasta 72 horas. Si nadie aprueba, se cancela automáticamente.
Despliegue en Kubernetes CD Tras la aprobación manual Aplica los manifiestos YAML al clúster. Actualiza las imágenes en los Deployments utilizando el SHA del commit. Kubernetes ejecuta un Rolling Update: reemplaza un pod a la vez, manteniendo el servicio disponible durante toda la actualización. Kubernetes detiene el rolling update. Los pods anteriores continúan atendiendo el tráfico.
Verificación de rollout CD Inmediatamente después del despliegue Espera hasta 3 minutos a que todos los pods estén en estado Running y superen sus verificaciones de salud. Si algún pod falla de forma repetida, Kubernetes lo detecta y el paso falla automáticamente. El pipeline falla con alerta. Es posible realizar un rollback manual con un único comando.

5 Integración con Kubernetes — Rolling Update

La estrategia de despliegue elegida es Rolling Update, que corresponde a la opción predeterminada de Kubernetes y la más adecuada para este sistema. En lugar de detener todos los pods y levantar los nuevos de forma simultánea —lo que causaría tiempo de inactividad—, Kubernetes reemplaza los pods de a uno mientras los demás continúan atendiendo peticiones.

5.1 Cómo ocurre un despliegue sin tiempo de inactividad

El backend tiene 3 réplicas configuradas. Cuando GitHub Actions actualiza la imagen, Kubernetes ejecuta la siguiente secuencia:

Estado inicial: 3 pods ejecutando la versión v1 Pod-1 [v1] ● en ejecución Pod-2 [v1] ● en ejecución Pod-3 [v1] ● en ejecución Paso 1: Kubernetes crea Pod-4 con la versión v2 Pod-1 [v1] ● en ejecución ↑ siguen atendiendo tráfico Pod-2 [v1] ● en ejecución Pod-3 [v1] ● en ejecución Pod-4 [v2] ◐ iniciando... Paso 2: Pod-4 supera la verificación de salud → Kubernetes termina Pod-1 Pod-4 [v2] ● en ejecución Pod-2 [v1] ● en ejecución Pod-3 [v1] ● en ejecución Pasos 3 y 4: se repite el ciclo hasta que todos los pods ejecutan v2 Pod-4 [v2] ● en ejecución Pod-5 [v2] ● en ejecución Pod-6 [v2] ● en ejecución ← despliegue completado, cero tiempo de inactividad

6 Gestión de secretos en el pipeline

Las credenciales nunca se almacenan en el repositorio. GitHub Actions proporciona un almacén de secretos cifrado, accesible únicamente durante la ejecución del workflow.

SecretDónde se utilizaDescripción
GITHUB_TOKEN Publicación Token automático de GitHub Actions. Permite subir imágenes al registro sin configuración adicional.
KUBE_CONFIG Despliegue Archivo kubeconfig del clúster codificado en base64. Permite a kubectl autenticarse y aplicar los manifiestos.

Las credenciales de la base de datos, las claves JWT y la configuración SMTP se gestionan directamente en Kubernetes como objetos Secret, separados del pipeline. El workflow de despliegue no tiene acceso a esas credenciales: únicamente las aplica al clúster.

7 Beneficios para StockAI SmartFlow

Velocidad de entrega

  • Un cambio en el backend puede estar en producción en menos de 10 minutos desde el merge
  • No hay pasos manuales que omitir ni ejecutar en orden incorrecto
  • Las 3 imágenes se construyen en paralelo usando la caché de GitHub Actions

Seguridad del despliegue

  • Ninguna versión con errores de sintaxis llega a producción
  • La etiqueta por SHA del commit permite identificar exactamente qué código está en ejecución
  • Si el rollout falla, los pods anteriores continúan activos de forma automática

Trazabilidad

  • Cada imagen Docker está etiquetada con el commit exacto que la generó
  • El historial de GitHub Actions registra quién desplegó qué y cuándo
  • El rollback requiere un único comando: actualizar la etiqueta al SHA anterior

Confiabilidad operacional

  • Kubernetes verifica que cada pod está sano antes de continuar el rolling update
  • La aprobación manual protege producción de cambios accidentales
  • El sistema de bodega nunca queda fuera de servicio durante un despliegue

Conclusión

La estrategia CI/CD propuesta transforma el despliegue de StockAI SmartFlow de un proceso manual propenso a errores en una cadena automatizada y auditable. GitHub Actions actúa como coordinador central: organiza la construcción, validación, publicación y despliegue de las tres imágenes del sistema de forma determinista. Kubernetes garantiza que cada actualización llega a producción sin interrumpir el servicio de bodega. La aprobación manual funciona como válvula de seguridad, preservando el control humano en el punto de mayor impacto sobre el negocio.
